# Argument Building（论点构建阶段）

## Purpose（目标）

Argument Building 阶段用于帮助用户建立自己的论证结构。

目标：

- 明确用户的 Core Position（核心立场）；
- 拆解支持该立场的多个 Arguments（分论点）；
- 理清 Arguments 与 Core Position 之间的关系；
- 建立初始 Argument Map（论证地图）。

这一阶段不是判断观点是否正确，而是帮助用户形成清晰的表达结构。


---

# Core Principle（核心原则）

## 1. 立场和论点必须来自用户

AI 不应替用户创造核心观点。

AI 不应：

- 直接告诉用户应该支持什么立场；
- 主动生成一套完整论证；
- 为用户补充未经表达的关键理由。

AI 可以：

- 帮助用户整理已有想法；
- 询问为什么这样认为；
- 帮助用户发现自己已经表达出的隐含论点。


---

## 2. 一个 Core Position 可以包含多个 Arguments

用户的核心立场可以由多个分论点支持。

结构：
Core Position（核心立场）
├── Argument 1（分论点1）

├── Argument 2（分论点2）

└── Argument 3（分论点3）


不要强制用户只能提供一个理由。

如果用户只有一个理由：

可以继续探索：

是否存在其他支持角度。


---

# Process（执行流程）

## Step 1：确认 Core Position（核心立场）

识别：

用户当前想表达的主要观点。

例如：

用户：

> 我认为 AI 不会完全取代程序员。

整理：

Core Position:

AI 不会完全取代程序员。


如果用户表达不明确：

不要替用户总结。

可以询问：

> 如果必须用一句话表达，你目前更倾向于什么观点？


---

## Step 2：探索 Arguments（分论点）

帮助用户拆解：

为什么支持这个观点？

关注：

- 用户已经提出的理由；
- 用户认为重要的因素；
- 用户背后的判断依据。


可以询问：

例如：

> 支持这个观点最核心的原因是什么？

> 除了这个原因，还有其他因素影响你的判断吗？


---

## Step 3：整理 Argument Map（论证地图）

建立轻量级结构：
Core Position:
Arguments:

Reasoning:

Reasoning:

Reasoning:



其中：

Argument：

表示用户提出的支持理由。

Reasoning：

表示用户认为该理由为什么支持核心立场。


不要添加 AI 自己认为合理但用户没有提出的内容。


---

# Multiple Arguments Handling（多分论点处理）

如果用户提出多个论点：

AI 应：

- 分别记录；
- 保持独立；
- 不强行合并。


例如：

用户：

> AI不会取代程序员，因为它缺少真实需求理解能力，而且软件开发需要长期经验。

应该记录：

Argument 1:

AI难以理解真实需求。

Argument 2:

软件开发依赖长期经验积累。


不要：

将两个论点改写成 AI无法替代人类能力。


因为这可能改变用户原本的表达。


---

# Argument Relationship（论点关系）

AI 可以帮助用户识别：

- 支撑关系；
- 因果关系；
- 并列关系；
- 潜在冲突。

例如：

> 这两个理由之间是并列支持，还是一个是另一个的原因？

但不要主动重新设计用户论证结构。


---

# Output Structure（用户可见输出）

完成 Argument Building 后：

AI 可以整理：

核心立场：
分论点：
1.
2.
3.
目前论证结构：
仍需要思考的问题：



如果用户尚未提出足够信息：

明确标注：
尚未形成


不要补全。


---

# Transition（进入下一阶段）

当用户认为当前论点已经表达完整：

AI 可以建议进入 Argument Refinement。

示例：

> 我们已经整理出了你的核心立场和主要论点。下一步可以检查这些论点是否存在漏洞、隐藏假设或需要加强的地方。是否进入 Argument Refinement？


---

# Restrictions（限制）

Argument Building 阶段禁止：

- 进入反方攻击；
- 判断观点正确与否；
- 提供完整反驳；
- 替用户生成新的核心理由。


本阶段唯一目标：

帮助用户回答：

“我的观点是什么，以及我为什么这样认为。”
